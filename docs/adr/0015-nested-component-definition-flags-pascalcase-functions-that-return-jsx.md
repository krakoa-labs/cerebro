# Nested component definition flags PascalCase functions that return JSX

Cerebro flags a Component for the nested-component footgun when its render body defines another component inline — the construct that creates a fresh component type on every parent render and so remounts the subtree, discarding its DOM nodes, focus, and state. The open question was what counts as "a component" among the functions a render body may declare, since flagging every inner helper would drown a lead in noise and flagging too narrowly would miss the real defect.

A nested component is recognized by two signals together: a **PascalCase name** and a **function that returns JSX**. This is React's own convention for distinguishing a component from an ordinary helper — `Row`/`Tab` are components, `renderRow`/`computeTotal` are not — and it is the same boundary the library itself relies on to decide what to call as a component. Requiring both keeps the indicator on the quiet side of ADR-0005: a PascalCase constant that is not a function (`const Total = 42`), a PascalCase function that returns no JSX, and a camelCase JSX-returning helper all stay unflagged. The "returns JSX" test stops at nested-function boundaries, so a non-component helper whose own result is non-JSX is never lifted by a JSX-returning function nested deeper inside it, and JSX produced only inside a `.map()` callback does not count as the surrounding function's own return. The render body is walked recursively, because the footgun hides as readily inside a conditional, a callback, or an effect as at the top of the body; the indicator is a boolean, so the walk short-circuits on the first nested component found. A `memo`/`forwardRef` wrapper around the inner function is seen through — a memoized nested component is the same footgun, worse — reusing the wrapper reader already shared by the other detectors.

The analysis is limited to function components reachable in the Component's own source. A class component's `render()` method is not traversed, and a component resolved from another module or an unparseable source is not analyzed; each falls to the quiet default. This matches every other source detector (ADR-0005) and avoids writing class-traversal machinery for a shape modern design systems have largely left behind.

## Considered Options

Recognizing styled-components and other CSS-in-JS factories as nested components was considered and deferred. A `styled.div\`…\`` defined in a render body is a genuine instance of this footgun — it builds a new component type each render exactly as a nested function does — but it is a non-function initializer (a tagged template or call), so the function-returning-JSX signal does not reach it, and it falls to the quiet default rather than being flagged. The validating design system uses no CSS-in-JS, so adding a `styled` reader now would be speculative machinery for a shape that does not occur; it is left until a concrete case demands it, at which point it is an additive change to the "is a component" signal, not a rewrite. The cost of deferring is bounded: the omission is in the quiet direction, never a false positive.

A name-only signal (flag any PascalCase declaration in the render body) and a returns-JSX-only signal (flag any inner function that returns JSX, regardless of name) were each rejected. Name-only false-positives on a PascalCase constant or a PascalCase non-component factory; returns-JSX-only false-positives on render-prop helpers and inline functions that legitimately produce JSX without being component definitions. Requiring both signals is what keeps the indicator precise enough to speak to a lead without manufacturing debt.
