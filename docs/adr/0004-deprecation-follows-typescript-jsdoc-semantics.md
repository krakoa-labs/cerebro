# Deprecation detection follows TypeScript JSDoc semantics

When marking a Component as deprecated, the canonical signal in the TypeScript/React ecosystem is a `/** @deprecated */` JSDoc tag on the declaration — that's the form IDEs and the TS language service surface as a strikethrough at usage sites. Cerebro mirrors that exactly: we read the `@deprecated` tag from the leading JSDoc of the declaration the public barrel export ultimately resolves to (or, for barrel-local declarations, the declaration in the barrel itself), and we ignore (a) non-JSDoc comment forms, (b) JSDoc on barrel re-export statements, and (c) deeply-wrapped patterns like `export default forwardRef(Foo)` where there is no traceable declaration. The result is that what Cerebro reports as deprecated lines up with what a developer sees as deprecated in their editor — no separate truth source to maintain.

Accepted consequence: teams that deprecate at the barrel boundary alone (e.g. to retire an alias without retiring the underlying declaration) will not see their Component flagged. If a real case emerges, this can be revisited; for now, strict alignment with TS semantics is the simpler invariant.
