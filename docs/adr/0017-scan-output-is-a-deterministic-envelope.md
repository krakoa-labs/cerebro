# Scan output is a deterministic envelope anchored to the committed state

A Scan emits its result not as the bare `components` array the CLI prints today, but wrapped in an envelope: a `schemaVersion` (the shape contract a consumer reads against), a `toolVersion` (the cerebro package version, used as a cache-staleness key — a persisted result produced by a different cerebro is stale and must be re-derived), the `scannedCommit` and its `committedAt`, the `config` snapshot, the per-Component `components`, the scan `warnings`, and the `git` availability (`warnings` and `git` are computed today but dropped at the CLI). The temporal field is `committedAt` — the committer date of `scannedCommit`, the same `%cI` primitive the Activity log already uses (ADR-0006) — and deliberately never a wall-clock "scannedAt". A wall-clock stamp would make the envelope non-deterministic (the same commit would yield different bytes on each run) and, more damagingly, defeat history-by-backfill: scanning twelve past commits today would stamp all twelve with today's date, collapsing them onto a single point on a trend's time axis, whereas each commit's committer date spreads them correctly. The envelope is therefore a pure function of (commit, config, toolVersion).

## Accepted consequence

A non-git design system has no commit to anchor to, so `scannedCommit` and `committedAt` are `null` and no history is possible there — consistent with git already being an environmental, gated capability (the Activity log, ADR-0006). The `git` block still records availability and shallowness so a consumer can tell a historyless result from a truncated one.
